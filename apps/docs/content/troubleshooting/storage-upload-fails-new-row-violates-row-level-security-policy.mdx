When a file upload to `POST /storage/v1/object/<bucket>/<path>` is rejected with a `400`/`403` whose body
contains `"new row violates row-level security policy for table \"objects\""` and your Postgres logs show
`COMMAND = INSERT`, `status = 42501`, this is Postgres correctly enforcing Row Level Security on
`storage.objects` — the `INSERT` did not satisfy a policy. It is **not** a Storage service malfunction.

Two details commonly cause confusion:

- **The log shows `USER = supabase_storage_admin`.** That is the *connection* role the Storage service logs
  in as. It owns the `storage` schema but does **not** have `BYPASSRLS` (hence `rolbypassrls = false`). For each
  request, Storage evaluates the `INSERT` under the **request's** role, derived from the token you send — `anon`
  if you send only the anon/publishable key, `authenticated` if you send a logged-in user's access token. So a
  policy written `TO supabase_storage_admin` does nothing for a normal upload; the policy that must pass is the
  one for `anon` or `authenticated`.
- **A "public" bucket does not exempt uploads from RLS.** Public only means objects have a public download URL;
  every other operation (including `INSERT`) still requires a matching policy. See
  [Why can't I upload/list/etc my public bucket?](/docs/guides/troubleshooting/why-cant-i-uploadlistetc-my-public-bucket-Z6CmGt).

## Why it happens

A `WITH CHECK` failure means no **permissive** `INSERT` (or `ALL`) policy passed for the effective role, **or**
a **restrictive** policy failed. The usual causes:

- **A `RESTRICTIVE` policy is present.** Permissive policies are combined using OR logic, meaning if any
  permissive policy passes the check succeeds. However, every `AS RESTRICTIVE` policy must *also* pass. A single
  restrictive policy that evaluates false blocks the insert no matter how many `WITH CHECK (true)` permissive
  policies you add. This is the most common reason checks still fail despite adding a policy for each case.
- **No permissive policy targets the effective role.** Policies created `TO authenticated` do not apply to an
  `anon` request, and vice versa. A `TO public` policy applies to all roles.
- **The policy is the wrong command or clause.** Only `FOR INSERT` / `FOR ALL` policies with a `WITH CHECK`
  expression gate inserts. A `FOR SELECT` policy, or a policy that only has `USING` and no `WITH CHECK`, does
  nothing for uploads.

## How to diagnose

**1. List every policy on `storage.objects` — look for restrictive ones and the effective role.**

Run this in the [SQL Editor](/dashboard/project/_/sql/new):

```sql
select policyname, permissive, roles, cmd, qual, with_check
from pg_policies
where schemaname = 'storage' and tablename = 'objects';
```

Any row with `permissive = 'RESTRICTIVE'` is a prime suspect. Confirm there is a `permissive = 'PERMISSIVE'`
row with `cmd` of `INSERT` or `ALL`, whose `roles` include the role you're uploading as (`anon` or
`authenticated`, or `{public}`), and whose `with_check` your row satisfies.

**2. Reproduce the upload the supported way and confirm the effective role.**

Always exercise the real upload path — the [Storage dashboard](/dashboard/project/_/storage/buckets), the
client library (`supabase.storage.from('candidate-cvs').upload(...)`), or the Storage API — rather than
inserting into `storage.objects` directly, which bypasses Storage's internal bookkeeping. Retry the upload as
the exact identity your app uses:

- **Anonymous** (only the anon/publishable key, no user token) → the request runs as `anon`.
- **Signed in** (a user access token in the `Authorization` header) → the request runs as `authenticated`.

Read the result against step 1:

- If step 1 has no `PERMISSIVE` `INSERT`/`ALL` policy matching that role (or a `RESTRICTIVE` policy that fails),
  the upload keeps returning `42501` — it's a policy issue on `storage.objects`. Fix the policies (next
  section).
- If step 1 shows a matching permissive policy and no failing restrictive one, but the upload still fails, the
  role/claims arriving on the request differ from what you expect — decode the token you send and confirm its
  `role`.

## How to fix

- **Remove or relax the restrictive policy** if step 1 found one, or rewrite it so legitimate uploads pass.
- **Add a permissive `INSERT` policy for the correct role.** A minimal working example:

  ```sql
  create policy "anon can upload to candidate-cvs"
  on storage.objects
  for insert
  to anon
  with check ( bucket_id = 'candidate-cvs' );
  ```

  Use `to authenticated` instead if uploads carry a user access token.
- **Drop the `TO supabase_storage_admin` policy** — it has no effect on normal uploads and is a red herring.

See [Storage access control](/docs/guides/storage/security/access-control) for more policy examples and
[Database API 42501 errors](/docs/guides/troubleshooting/database-api-42501-errors) for the general error.

## How to prevent

Configure `storage.objects` policies before going to production, and test uploads as both `anon` and
`authenticated` (whichever your app uses). Avoid adding `RESTRICTIVE` policies unless you specifically intend
to AND-gate every operation.
