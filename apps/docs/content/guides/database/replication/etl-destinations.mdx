## Choosing a Destination

### BigQuery vs Iceberg

| Feature | BigQuery | Iceberg (Analytics Buckets) |
|---------|----------|----------------------------|
| **Setup complexity** | Simple (GCP service account) | Simple (S3 credentials) |
| **Query engines** | BigQuery SQL | Spark, Trino, Presto, DuckDB, etc. |
| **Storage** | Google Cloud Storage (managed) | S3-compatible (managed by Supabase) |
| **Cost model** | Pay per query + storage | Storage-based |
| **Time travel** | Limited retention | Full Iceberg support |
| **Best for** | Google Cloud ecosystem | Open-source data lakes |
| **CDC metadata** | Included in tables | Available in Iceberg format |

### Use Case Recommendations

**Choose BigQuery if you:**
- Are already using Google Cloud Platform
- Need built-in business intelligence tools
- Want a fully managed query experience
- Require tight integration with Google services

**Choose Iceberg if you:**
- Want an open table format
- Need to query data with multiple tools (Spark, Trino, etc.)
- Are building a data lakehouse architecture
- Want full control over query engines
- Need advanced time travel capabilities

## Getting Started

To set up a destination:

1. **[Create a publication](/docs/guides/database/replication/etl-replication-setup#step-1-create-a-publication)** - Define which tables to replicate
2. **Choose your destination** - Select BigQuery or Iceberg based on your needs
3. **Configure the destination** - Follow the destination-specific setup guide
4. **Monitor replication** - Track status and performance

## Common Features

All ETL destinations provide:

- **Automatic schema creation**: Tables are created automatically to match your Postgres schema
- **Change data capture**: INSERT, UPDATE, and DELETE operations are tracked
- **Batched delivery**: Changes are sent in optimized batches for performance
- **Error handling**: Failed changes are automatically retried
- **Built-in monitoring**: Track lag, throughput, and errors in the Dashboard

## Limitations

Some limitations apply to all destinations:

- **Primary keys required**: Tables must have primary keys
- **TOAST values**: Large values limited to 10MB
- **Custom data types**: Not supported
- **Event filtering**: All event types (INSERT, UPDATE, DELETE) are required
- **Schema changes**: No automatic support for schema alterations

For destination-specific limitations, see the individual destination pages.
