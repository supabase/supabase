## Integrating with Supabase Auth

Use the [`@supabase/server`](https://github.com/supabase/server) package to secure your Edge Functions. It wraps your fetch handler, verifies the caller's JWT, and hands you a pre-configured Supabase client.

With `withSupabase({ allow: 'user' })` you get:

- JWT verification against Supabase JWKS, done locally with no network call
- `ctx.supabase`, scoped to the caller and respecting Row Level Security
- `ctx.supabaseAdmin`, which bypasses Row Level Security for privileged work
- CORS preflight and response headers, handled for you
- An automatic 401 JSON response when the token is missing or invalid

Because `allow: 'user'` expects a valid JWT, you can leave the platform-level `verify_jwt` flag enabled (the default). Disable it in `supabase/config.toml` only if you switch to `allow: 'public'`, `allow: 'secret'`, or `allow: 'always'`.

```ts
import { withSupabase } from 'npm:@supabase/server'

export default {
  fetch: withSupabase({ allow: 'user' }, async (_req, ctx) => {
    const email = ctx.userClaims?.email
    return Response.json({ message: `hello ${email}` })
  }),
}
```

If you want to shape the error response yourself, use `createSupabaseContext` instead. It returns a `{ data, error }` tuple so you stay in control of the response:

```ts
import { createSupabaseContext } from 'npm:@supabase/server'

export default {
  fetch: async (req: Request) => {
    const { data: ctx, error } = await createSupabaseContext(req, {
      allow: 'user',
    })
    if (error) {
      return Response.json(
        { message: error.message, code: error.code },
        { status: error.status }
      )
    }
    return Response.json({ message: `hello ${ctx.userClaims?.email}` })
  },
}
```

## Accepting both users and services

Some functions need to answer both authenticated end users and internal services like cron jobs, backend workers, or other Edge Functions. Pass an array to `allow` to accept multiple auth modes, then branch on `ctx.authType` inside the handler.

- Modes are tried in order. The first match wins.
- `ctx.authType` tells you which mode matched (`'user'` or `'secret'`).
- Use `ctx.supabase` for user calls (RLS-scoped) and `ctx.supabaseAdmin` for service calls (bypasses RLS).
- A request that satisfies neither mode gets a 401 automatically.

```ts
import { withSupabase } from 'npm:@supabase/server'

export default {
  fetch: withSupabase({ allow: ['user', 'secret'] }, async (req, ctx) => {
    if (ctx.authType === 'user') {
      const { data } = await ctx.supabase.from('reports').select()
      return Response.json(data)
    }

    const { userId } = await req.json()
    const { data } = await ctx.supabaseAdmin
      .from('reports')
      .select()
      .eq('user_id', userId)
    return Response.json(data)
  }),
}
```

Because this function can be called without a user JWT, set `verify_jwt = false` for it in `supabase/config.toml`. Otherwise the platform rejects service calls before they reach your handler.

```toml
[functions.reports]
verify_jwt = false
```

## Service-to-service calls

Some functions are called by internal services instead of end users: cron jobs, another Edge Function, a backend worker, or `pg_net` from Postgres. Gate these endpoints with a named secret key so only the intended caller can reach the function.

`allow: 'secret:<name>'` validates the `apikey` header against the specific named key in `SUPABASE_SECRET_KEYS`. Requests without that exact key get a 401.

Because the caller isn't sending a user JWT, set `verify_jwt = false` for the function in `supabase/config.toml`.

```toml
[functions.run-automations]
verify_jwt = false
```

```ts
import { withSupabase } from 'npm:@supabase/server'

export default {
  fetch: withSupabase({ allow: 'secret:automations' }, async (req, ctx) => {
    const { taskId } = await req.json()
    const { data } = await ctx.supabaseAdmin
      .from('scheduled_tasks')
      .update({ status: 'done' })
      .eq('id', taskId)
      .select()
      .single()
    return Response.json(data)
  }),
}
```

The caller sends the named secret key in the `apikey` header:

```ts
await fetch('https://<project-ref>.supabase.co/functions/v1/run-automations', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    apikey: Deno.env.get('SB_SECRET_AUTOMATIONS')!,
  },
  body: JSON.stringify({ taskId: 'task_123' }),
})
```

### External webhooks

External providers like Stripe, GitHub, or Linear can't send Supabase API keys. They send their own signed payload instead. For these endpoints, use `allow: 'always'` to skip the SDK's credential check, then verify the provider's signature inside the handler. See the [`@supabase/server` package docs](https://github.com/supabase/server) for a full Stripe webhook example.

<Admonition type="caution">

`allow: 'always'` disables all credential checks. Your handler is fully responsible for authenticating the caller. Never use it on an endpoint that reads or writes sensitive data without verifying the caller some other way, such as a signed webhook payload.

</Admonition>

## Environment variables

`@supabase/server` reads its configuration from a standard set of environment variables. On the Supabase platform and in local development with the CLI, the required variables are auto-provisioned, so you don't need to set anything.

| Variable                    | What it is                                 |
| --------------------------- | ------------------------------------------ |
| `SUPABASE_URL`              | Your project URL                           |
| `SUPABASE_PUBLISHABLE_KEYS` | Named publishable keys as a JSON object    |
| `SUPABASE_SECRET_KEYS`      | Named secret keys as a JSON object         |
| `SUPABASE_JWKS`             | JSON Web Key Set used to verify user JWTs  |

Local development with the CLI uses a single-key setup, which the SDK also accepts as a fallback: `SUPABASE_PUBLISHABLE_KEY` and `SUPABASE_SECRET_KEY`.


<Admonition type="note">

The same zero-config experience is available on other runtimes. Install [`@supabase/server`](https://github.com/supabase/server) in your Node.js, Bun, Cloudflare Workers, or self-hosted Deno app and set the required environment variables. See the package's [environment variables guide](https://github.com/supabase/server/blob/main/docs/environment-variables.md) for the full reference.

</Admonition>
